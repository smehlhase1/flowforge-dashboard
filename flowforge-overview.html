Claude Code · spec-driven development

FlowForge

A structured pipeline inside Claude Code that covers the full software feature lifecycle — from initial problem framing through to a verified, CI-delivered PR. Today (v1.1.1), 3 agents support the BA spec phase and 14 agents cover the dev pipeline. The BA Upstream Pipeline (CIL-6148) extends tool coverage to the 6–10 hours of upstream work that currently happens without any agent support, adding 5 commands and 15+ agents to close that gap end-to-end.

The full pipeline — available today

Nine commands carry a feature from pre-flight through advisory review. The BA owns requirements; the developer drives planning through review. Each <feature-id> threads one feature through the whole run.

Supporting commands

Utility commands that work independently of the main pipeline — or enhance it at any stage.

Start
/flowforge.start <feature-id>

Pre-flight before any spec work. Pulls latest master, guards against branch dependency traps, creates the feature branch.

Run once per feature before /flowforge.require. Stops with a clear message if master has diverged or if open local branches may create implicit dependencies.

Investigate
/flowforge.investigate "<question>"

Answer a codebase question with file-line citations. Works before any feature exists — no pipeline state required.

Read-only. Returns: answer, evidence (path:line), open questions if evidence is incomplete. Use instead of starting a full feature for read-only queries.

Replay
/flowforge.replay <id> --from <step>

Rerun the pipeline from a selected step onwards using the current spec and artifacts.

Steps: plan · analyze · build · verify. Reruns the selected step and all downstream steps — no smart diff.

Status
/flowforge.status

Show a table of all features — pipeline state, spec/plan/verify artifact presence, and next likely step.

Read-only. Flags failed and fixing features with a brief note pulled from verification-report.md.

Init
/flowforge.init

Initialize FlowForge in the project — creates specs/, .flowforge/, lane config, and optional Jira integration.

Run once per project before using any other FlowForge command.

The four lanes

Express and Fast go through JIRA automation; Standard and Heavy follow the normal process. The split is about how obvious the design is — not how big the change feels.

Lane 01

Express

⚡ Automated

Repeatable, well-patterned foundational work that follows established conventions.

Looks likeNew partner integration scaffolds · CRUD endpoints from an established pattern · multi-service config for a new environment.
👤1 human gate
Lane 02

Fast

⚡ Automated

Small-scope corrections within established patterns — a few hours of work.

Looks likeAdapter fix for a changed API contract · a business-rule or validation tweak · add a single field to an endpoint.
👤1 human gate
Lane 03

Standard

Normal process

Core feature work with novel business logic the design isn't obvious for.

Looks likeNew business logic we haven't built before · changes touching multiple services in non-obvious ways.
👤👤👤3 human gates
Lane 04

Heavy

Classification only

Complex, exploratory, or architectural work needing real engineering judgment. Classification only — no Heavy pipeline is available.

Looks likeArchitecture decisions & design spikes · major refactors · bug investigations with an unknown cause.
👤👤👤👤4+ human gates
⚡ What "automated" means

A well-scoped JIRA ticket labelled express or fast moves to AI Generation Ready; ~15–25 min later a draft PR links back to the ticket. The engineer reviews it like a junior's PR, tests it, refines it — and owns it. Nothing merges without human review.

Rule of thumb

One sitting following existing patterns → Express or Fast. Non-obvious design decisions → Standard or Heavy. When in doubt, default to Standard — and any agent or engineer can escalate a lane up when reality disagrees.

What runs in each lane

The shared pipeline, dialled per lane. Automated lanes lean on JIRA + agents with one human sign-off; normal-process lanes add review and approval gates. Human grooming (require) runs in every lane.

Traceability from first thought to CI delivery

Once CIL-6148 ships, every artifact flows into the next step: 0-discovery.md → 0-research.md → 0-options.md → 1-spec.md → 0-split-map.md → 0-refine-agenda.md → 0-meeting-log.md → 0-prd.md → 2-plan.md → code → CI. The acceptance criteria a BA discusses in /ff.discover are the same criteria the verifier checks in /ff.verify. Nothing gets lost in translation — and nothing bypasses the chain. status.json tracks the full state: start_confirmed, start_branch, jira_key, jira_sub_keys, confluence_url, phase, last_updated.

The complete pipeline — mandatory vs optional

Across both phases — BA upstream and dev — only five BA upstream steps plus the dev pipeline are required to complete a feature end-to-end. /ff.discover, /ff.refine, and /ff.prd are optional and additive — the dev pipeline is always reachable from /ff.start + /ff.require alone.

Jira tickets — /ff.require (enhanced)

/flowforge.require enhanced creates Jira tickets at the end of Phase 5 — after spec capture, 8-agent review, decomposition, and agenda building are complete. Each ticket has 4 sections: Refinement Brief (Why/What) · Solution Options · ACs · Refinement Agenda (with numbered items from 0-refine-agenda.md). Decomposition sub-specs (Phase 4) create sub-tickets linked under the parent. Ticket creation is user-confirmed; Jira hierarchy is enforced: theme → initiative → epic → task.

Jira sync — /ff.refine

/flowforge.refine (live facilitation) updates existing Jira tickets after each agenda item is resolved. The Jira Sync agent enforces hierarchy — theme → initiative → epic → task — and refuses to create parentless tickets below theme level. All writes are user-confirmed. The meeting log is stored in 0-meeting-log.md and decisions are applied to 1-spec.md by the Spec Editor agent.

Minimum viable run — two commands, full pipeline reachable

The entire BA upstream phase is additive. A feature can go from zero to a pushed spec and Jira ticket with just two commands:

/flowforge.start <feature-id>  →  branch created · status.json initialised
/flowforge.require <feature-id>  →  5 phases · spec · split-map · agenda · Jira tickets · pushed to origin

The three optional upstream commands (.discover · .refine · .prd) add depth and traceability, but the dev pipeline is fully reachable from just those two. The richer the upstream, the fewer surprises in .plan and .verify.

Now (v1.1.1): start → require → lane → plan → analyze → build → verify → fix → review, with /flowforge.express and /flowforge.fast running full pipelines in a single command. QA commands: qa.manual · qa.automation · qa.scenarios-from-history. Supporting: start · investigate · replay · status · init.

Future (CIL-6148): start (patch) → discover [3 phases · 10 agents] → require enhanced [5 phases · 8 agents] → refine [live facilitation] → prd [13 sections · living doc] — 1 patched command + 4 new commands, adding 15+ agents for the upstream BA phase.
