What is FlowForge?

FlowForge is a multi-agent pipeline that runs inside Claude Code and covers every stage of a feature's life — from the BA's first problem framing session, through spec capture and refinement, into code generation via TDD, and out through a gated evidence-based review before peer code review starts. BAs stay in charge of requirements; engineers stay in charge of code — agents draft, humans decide, and nothing merges without explicit human review.

Today (v1.1.1) the dev pipeline is live with 14 agents. CIL-6148 adds a 5-command BA upstream pipeline so FlowForge starts at problem framing, not spec capture. PROD-13861 extends /flowforge.review from advisory output into a gated IIE pipeline that replaces the solo human pre-analysis step — so the first human touch in review is peer code review, not solo triage.

What changes

Today, /flowforge.review produces advisory AI feedback only — no gate, no pipeline. After PROD-13861, it becomes a gated, evidence-driven replacement for the solo human code analysis step. A single Independent Implementation Examiner (IIE) agent covers all review dimensions, deterministic tooling runs before the IIE starts, and a policy routing engine decides what to auto-fix versus escalate. The 2-approver peer review is not eliminated — only the preparatory solo analysis phase is automated.

Gate verdicts

PASS — zero critical, zero undismissed high, 100% AC traceability. All three simultaneously required.  |  PASS_WITH_FINDINGS — low/medium record-only findings only; PR can open.  |  FIX_REQUIRED — auto-fixable findings; max 2 cycles before escalation.  |  ESCALATE — architecture/compliance/GDPR findings, or fixLoopCount reached 2 with remaining critical/high. Human decision required.  |  INSUFFICIENT_EVIDENCE — IIE could not substantiate confidence on key dimensions.

The four lanes

Express and Fast go through JIRA automation; Standard and Heavy follow the normal process. The split is about how obvious the design is — not how big the change feels. /ff.review runs in all four lanes.

Lane 01

Express

⚡ Automated

Repeatable, well-patterned foundational work following established conventions.

Looks likeNew partner integration scaffolds · CRUD endpoints from an established pattern · multi-service config for a new environment.
👤1 human gate
Lane 02

Fast

⚡ Automated

Small-scope corrections within established patterns — a few hours of work.

Looks likeAdapter fix for a changed API contract · a business-rule or validation tweak · add a single field to an endpoint.
👤1 human gate
Lane 03

Standard

Normal process

Core feature work with novel business logic the design isn't obvious for.

Looks likeNew business logic we haven't built before · changes touching multiple services in non-obvious ways.
👤👤👤3 human gates
Lane 04

Heavy

Classification only

Complex, exploratory, or architectural work needing real engineering judgment. Classification only — no Heavy automated pipeline.

Looks likeArchitecture decisions & design spikes · major refactors · bug investigations with unknown cause.
👤👤👤👤4+ human gates
⚡ What "automated" means

A well-scoped JIRA ticket labelled express or fast moves to AI Generation Ready; ~15–25 min later a draft PR links back to the ticket. The engineer reviews it like a junior's PR, tests it, refines it — and owns it. Nothing merges without human review.

Rule of thumb

One sitting following existing patterns → Express or Fast. Non-obvious design decisions → Standard or Heavy. When in doubt, default to Standard — any agent or engineer can escalate a lane up when reality disagrees.

What runs in each lane

The shared pipeline, dialled per lane. Automated lanes lean on JIRA + agents with one human sign-off; normal-process lanes add review and approval gates. /ff.review IIE gate runs in every lane.
