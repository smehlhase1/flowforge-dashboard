What is the Integration Layer?

The Integration Layer (IL / CIL) is Simplesurance's "One API" platform — partners integrate once and access multiple insurance products in multiple countries. It abstracts heterogeneous Allianz backends behind a single, well-documented REST API.

Strategic purpose
  • Single REST API for partners — no per-country, per-LoB integrations
  • XLoB: Travel, Motor, Property, RSA, Health in one layer
  • Pan-European + APAC reach through unified routing
  • Hosted on AWS Cloud, microservices, Kubernetes, Temporal for workflow orchestration
  • OAuth2 / OIDC auth, mTLS, PCI DSS & GDPR by design

Links insured persons to an existing travel policy and contract. Payload includes product_code, policy_number, contract details, and an array of beneficiaries with personal data and contact info.

Returns 201 Created — no body. All calls are asynchronous.
Routing
All requests route to BIOS. product_code identifies the market/product context within BIOS.

Contract Management Endpoint — Deep Dive

Standardized post-sale policy lifecycle API. Product-agnostic: operates on core entities like policy_number and quote_id, applicable across Travel, Motor, Health — any LoB integrated with CIL.

Add / Update / Remove insured person
POST/policies/{policy_number}/insured-person
PUT/policies/{policy_number}/insured-person
DELETE/policies/{policy_number}/insured-person/{person_id}
Use cases:
Add new member to health/travel policy · Update contact details · Remove a traveller from a policy

Response: 200 OK with person_id (for POST) · 404 if policy/person not found
Key fields
FieldNotes
policy_numberPath param — routes to backend
person_idReturned on POST, required for PUT/DELETE
first/last_name, DOBCore person data
address.countryISO 3166-1 alpha-2
contact.email / phoneEither/both required for holder
Cancellation eligibility check
POST/policies/{policy_number}/cancellation-eligibility
Checks whether a policy is eligible for cancellation based on refund type and cancellation reason. Returns eligible: true/false, cancellation_date, and a message.

Partners use this before offering a cancellation option in the UI — prevents invalid cancellation attempts and reduces backend errors.
Sample response
{
  "eligible": true,
  "cancellation_date": "2025-11-03",
  "message": "Cancellation is eligible."
}
Request body: refund_type (PRORATED etc.) + cancellation_reason
Get & action a saved quote
GET/quotes/{quote_id} — retrieve full quote
POST/quotes/{quote_id}/quote-actions
GET: Full quote including trip details, destinations, premium, product name, coverages, and purchasability status.

Quote Actions: event_type SAVE_QUOTATION — sends the quote by email to the customer. Supports agent_email for agent-driven flows.

Returns 204 No Content on success.
Quote action payload
{
  "event_type": "SAVE_QUOTATION",
  "email_recipients": {
    "to": ["customer@example.com"]
  },
  "customer": {
    "customer_first_name": "John",
    "customer_last_name": "Doe"
  },
  "agent_data": {
    "agent_email": "agent@partner.com"
  }
}
Document upload
POST/policies/upload-documents
Uploads documents (e.g. medical questionnaires, signed forms) as multipart/form-data and associates them with policies or persons via a correlations array.

Header: X-Sisu-Request-ID (UUID) — idempotency key.

Returns document_id per uploaded file, plus success/failure status per document.
Correlation types
correlation.typeUse case
PERSON_IDAssociate doc with insured person
POLICY_NUMBERAssociate with policy directly
Primary usage: medical screening documents in Fox UK / Rush/Hays flows (Verisk integration)
Claims Endpoint — Deep Dive
