What is the Integration Layer?

The Integration Layer (IL / CIL) is Simplesurance's "One API" platform — partners integrate once and access multiple insurance products in multiple countries. It abstracts heterogeneous Allianz backends behind a single, well-documented REST API.

Strategic purpose
  • Single REST API for partners — no per-country, per-LoB integrations
  • XLoB: Travel, Motor, Property, RSA, Health in one layer
  • Pan-European + APAC reach through unified routing
  • Hosted on AWS Cloud, microservices, Kubernetes, Temporal for workflow orchestration
  • OAuth2 / OIDC auth, mTLS, PCI DSS & GDPR by design

Beneficiaries Endpoint — Deep Dive

Manages the full beneficiary lifecycle for travel insurance policies — onboarding, modification, and offboarding — via BIOS, the Allianz open-policy admin backend. CIL acts as middleware; BIOS is the single system of record for beneficiary data across all markets.

Data flow — all scenarios
Onboarding
Partner / Frontend
→ POST /beneficiaries
CIL
validates · routes by product_code · transforms
→ BIOS: creates beneficiary record · returns 201 Created
Modification
Partner / Frontend
→ PUT /beneficiaries/{id}
CIL
validates beneficiary_id · routes · updates payload
→ BIOS: updates personal data / contact info · returns 200 OK
Offboarding
Partner / Frontend
→ DELETE /beneficiaries/{id}
CIL
validates · routes beneficiary removal
→ BIOS: removes beneficiary record · returns 204 No Content
⚠ All calls are asynchronous — requests pending backend approval
Endpoints
POST/beneficiaries → onboard one or more beneficiaries
PUT/beneficiaries/{id} → modify beneficiary data
DELETE/beneficiaries/{id} → offboard beneficiary
Payload includes product_code, policy_number, contract details, and beneficiary personal data & contact info.
Routing
All requests route to BIOS. product_code identifies the market/product context within BIOS.
Backend: BIOS
Beneficiaries are managed through BIOS — the Allianz open-policy admin backend that handles beneficiary records across all markets.

BIOS is the single system of record for beneficiary data. CIL translates the incoming request and routes it to BIOS regardless of country.
Markets covered: UK · DE · HU · BE · NL · LU · PL · RO · ES · FR · IT · PT

Contract Management Endpoint — Deep Dive

Post-sale policy lifecycle API. Routes through public-apis/simplesurance → travel-service (gRPC) → carrier adapters.

Endpoints
POST/policies/{policy_number}/cancellation-eligibility
PATCH/policies/{policy_number} — update policy
PUT/policies/{policy_number}/replace
POST/policies/v1/{policy_number}/amendments
POST/policies/v1/{policy_number}/amendments/{id}/accept
GET/policies/v1/{policy_number}/capabilities
Cancellation eligibility — request body: cancellation_reason + refund_type (PRORATED / FULL / NO_REFUND). Response: eligible, cancellation_date, message.

PATCH /policies — updatable fields: end_date, holder, additional_data (beneficiaries, trip details, auto_renewal, refund_type), effective_date, contract_status.

Amendments — POST creates amendment. Accept routes to travelService.AcceptPolicyAmendment() → EAPI POST /modifications/{id}/confirmation. Response: 200 AWAITING_PAYMENT (returns session_id) or 202 CONFIRMATION_ACCEPTED.

Upload documents endpoint
POST/policies/upload-documents — multipart/form-data
Header: X-Sisu-Request-ID (required, UUID — idempotency key).

Size limits: 10 MB per file · 50 MB total.

Field naming: sequential index — documents[0]key, documents[0]file, documents[0]correlations, documents[0]document_type, documents[0]metadata.

Routing: Motor insurance type → motorService.UploadDocument(). Travel (default) → travelService.UploadDocument() (streaming gRPC — metadata sent first, then file bytes).

Response: array of {key, correlations, status: SUCCESS|FAILURE, result: {document_id}}.

Routing logic
CIL determines the backend from a combination of partner_id, country, and product_code from the request, matched against per-partner flow configuration stored in CIL.

Each partner has an isolated flow config — changing one partner's routing does not affect others.

Multi-backend markets (e.g. CA uses both eMagin + Fusion) are handled by Temporal workflow orchestration, which coordinates the calls and compensates on failure.
